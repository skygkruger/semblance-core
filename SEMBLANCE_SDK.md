## What the SDK Is

The Semblance SDK is not a set of API wrappers around Semblance features. It is Semblance's sovereign architecture — the two-process isolation model, the Merkle audit chain, the autonomy framework, the knowledge graph — packaged as a deployable primitive that any developer can build on.

A developer building a healthcare app, a legal tool, a personal finance product, or any application that handles sensitive user data can depend on the SDK and inherit Semblance's architectural guarantees. They don't build privacy infrastructure. They inherit it. Their app can make the same architectural claims Semblance makes because they are using the same architecture, enforced by the same mechanisms.

The wrong frame: "API for Semblance features."  
The right frame: "Sovereign AI architecture as a primitive."

## The Core Proposition for Developers

### 1. Two-Process Isolation as a Dependency

The AI Core / Gateway split with all its enforcement — OS sandbox, import-level CI audit, Tauri capability declarations — deploys as a dependency. The developer's AI logic runs in the Core. Their external service integrations run in the Gateway. The isolation is enforced by the SDK infrastructure, not by the developer's discipline.

```typescript
import { SemblanceCore, SemblanceGateway } from '@veridian/semblance-sdk'

// Developer's AI logic runs in isolation — zero network access enforced
const core = new SemblanceCore({
  appId: 'com.example.myapp',
  knowledgeGraph: true,
  autonomyFramework: true,
})

// Developer's external integrations run through audited Gateway
const gateway = new SemblanceGateway({
  allowlist: ['api.stripe.com', 'api.twilio.com'],
  auditChain: true,
})
```

The developer writes their application logic. The sovereignty invariants are enforced by the SDK.

### 2. The Audit Chain as an API

Any action their app takes can be logged to a tamper-evident Merkle chain. Not Semblance's audit chain — the user's. The user's audit trail becomes a unified record of everything every Semblance-based app has ever done on their behalf.

```typescript
// Every external action logs automatically when routed through Gateway
await gateway.execute({
  type: 'email.send',
  to: 'client@firm.com',
  subject: 'Matter update',
  body: draft,
  // Automatically: signed, logged before execution, logged after
})
```

One sovereignty report covers all SDK apps the user has installed. The user doesn't need to trust each app individually — they verify the audit chain.

### 3. Knowledge Graph Contribution with Consent

SDK apps can contribute data to the user's knowledge graph with explicit consent UI generated by the SDK — not the app. The developer declares what they want to contribute. The SDK presents a standardized consent flow to the user. The app receives write access after approval.

```typescript
// Developer declares intent
const contribution = sdk.requestKnowledgeGraphAccess({
  dataTypes: ['listening_history', 'mood_tags'],
  description: 'Your Spotify listening history helps Semblance understand your schedule and recommend music for different contexts.',
  revocable: true, // always true — SDK enforces this
})

// SDK presents standardized consent UI — not the app's UI
// User sees: what data, why, how it's used, how to revoke
// After approval:
await contribution.write({ source: 'spotify', data: listeningHistory })
```

The user gets one unified knowledge graph. Developers get access to the most powerful personal data substrate available. Trust is maintained because the consent flow is standardized and the SDK enforces revocation.

### 4. Autonomy Framework as a Decision API

SDK apps query the user's autonomy framework before taking actions. The user sets their trust level once. Every SDK app respects it consistently.

```typescript
// App asks before acting — user's tier determines the answer
const decision = await sdk.autonomy.decide({
  actionType: 'email.send',
  context: { recipient: 'client@firm.com', urgency: 'low' }
})

if (decision === 'auto_approve') {
  await send()
} else if (decision === 'requires_approval') {
  await presentForApproval(draft)
} else {
  // blocked — log and surface reason
}
```

### 5. Gateway Allowlist Inheritance

SDK apps route their external network calls through the user's Gateway. They inherit domain allowlist enforcement, rate limiting, anomaly detection, and audit logging automatically. A developer builds none of that infrastructure — it comes with the SDK.

## What the SDK Is Not

- Not a cloud API. The SDK runs on the user's device. There are no VERIDIAN servers in the data path.
- Not a Semblance feature toggle. SDK apps are fully independent applications that share the sovereign architecture.
- Not a way to access other users' data. Every user's knowledge graph is isolated. Cross-user data requires explicit Semblance Network consent from both parties.
- Not a way to bypass sovereignty. SDK apps are subject to the same five inviolable rules as Semblance itself. The CI audit, OS sandbox, and Gateway enforcement apply to SDK apps.

## The Network Effect

Every developer who builds on the SDK becomes a node in a network where user data stays sovereign. Every SDK app that contributes to the user's knowledge graph makes Semblance more powerful for that user. Every user whose knowledge graph grows through SDK app contributions has stronger reason to stay in the Semblance ecosystem.

The SDK is not a monetization play at launch. It is a moat-building play. The more SDK apps exist, the more powerful the user's unified knowledge graph becomes, and the stronger Semblance's position becomes relative to any cloud alternative.

## Moat Implications for VERIDIAN

The SDK creates a secondary moat alongside Semblance's primary consumer moat:

**Consumer moat:** knowledge graph compounds on user's device — cloud competitors cannot catch up without the historical data.

**SDK moat:** developer ecosystem built on sovereign architecture — app developers who build on SDK are building on an architectural premise that locks them out of cloud AI backends. Their privacy claims depend on the SDK continuing to deliver. Switching away from the SDK means rebuilding their privacy story from scratch.

## Build Sequence

The SDK ships after the consumer app is stable and the architecture is proven in production. Attempting to build the SDK before the consumer architecture is locked risks building the wrong abstraction layer.

**Prerequisites before SDK work begins:**
- Consumer app shipped and in production
- Hardware Bridge complete and stable
- Tunnel infrastructure complete and stable
- Sovereign fine-tuning architecture reservation validated
- At least 3 months of consumer production data to inform the right API surface

**Estimated timeline:** 6–9 months post consumer launch.
